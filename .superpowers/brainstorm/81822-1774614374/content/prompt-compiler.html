Section 2: Prompt Compiler Engine

The core engine that turns prompt packs into compiled system prompts per request

Prompt Compilation Flow
User Message
+ conversation history
Team Config
pack ID, model, mode
Documents
uploaded + retrieved chunks
▼
STEP 1: CLASSIFY REQUEST
• Keyword matching against module use_when tags
• Detect task mode (analysis, implementation, tender, etc.)
• Check if task form was used → mode is pre-selected
▼
STEP 2: SELECT MODULES
Layer A
Always loaded
Identity, reasoning, execution, output rules
Layer B
By classification
Domain modules matched to request
Layer C
Per request
Doc chunks, task context, constraints
▼
STEP 3: COMPILE SYSTEM PROMPT
system_prompt =
  safety_wrapper (hardcoded, highest priority)
  + core_files (Layer A from manifest)
  + always_append (from manifest)
  + domain_modules (Layer B, matched)
  + mode_prompt (analysis/implementation/etc.)
  + doc_context (Layer C, injected or retrieved)
  + precedence_rules (conflict resolution)
▼
STEP 4: TOKEN BUDGET CHECK
• Count tokens against model's context limit
• If over budget: trim doc chunks first, then summarize domain modules
• Reserve tokens for conversation history + response
• Log final composition for debugging
▼
Send to LLM Provider
system prompt + messages → stream response via SSE

Prompt Pack Data Model

How packs are stored and managed
DB: prompt_packs table
id, name, version
team_id (owner)
manifest (JSON — core, domains, always_append)
created_at, updated_at
DB: prompt_modules table
id, pack_id (FK)
filename, title
layer (core / domain / always)
tags[] (use_when keywords)
priority, content (markdown text)
max_tokens
DB: task_modes table
id, pack_id (FK)
name (analysis, implementation, ...)
prompt_text
form_schema (JSON — for task forms)

Key Points

  • Manifest lives in DB — admins edit via UI, but can also import from JSON file
  • Module frontmatter (tags, priority, max_tokens) is parsed on upload and stored as columns for fast querying
  • Classification is keyword-first — fast and predictable. Can add LLM-based classification later as an upgrade
  • Token budgeting prevents context overflow — trims least-priority content first
  • Compiled prompts can be cached per (pack_version + mode + matched_domains) tuple
