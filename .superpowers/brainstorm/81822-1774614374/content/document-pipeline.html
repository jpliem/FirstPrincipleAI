Section 3: Document Pipeline & Chat Flow

How documents are processed and how chat requests flow through the system

Document Upload & Processing Pipeline
User uploads file
PDF, DOCX, TXT, CSV, images
▼
API: Upload Endpoint
• Save file to storage (local / S3)
• Create documents record (status: pending)
• Dispatch background task → return immediately
▼
BACKGROUND WORKER
1. Parse
PDF → PyMuPDF / pdfplumber
DOCX → python-docx
Images → OCR microservice
CSV/TXT → direct read
2. Decide Strategy
Token count < threshold?
→ SMALL: store full text
→ LARGE: chunk + embed
3. Chunk & Embed
Split by sections/paragraphs
Embed via model's embeddings API
Store vectors in pgvector
Keep chunk ↔ doc mapping
Update document status: pending → processing → ready / failed
▼
Document ready for chat
Small docs: full text cached | Large docs: vectors indexed
Chat Request Flow
1
User sends message
Text + optional attached doc IDs
2
Load conversation history
From DB, trim to fit token budget
3
Retrieve document context
Small docs → inject full text
Large docs → pgvector similarity search → top-k chunks
4
Prompt Compiler
Classify → select modules → compile system prompt → budget check
5
Call LLM Provider
system_prompt + messages → stream via team's configured model
6
Stream response via SSE
Tokens streamed to frontend as they arrive
Full response saved to DB on completion

Key Points

  • Upload is non-blocking — API returns immediately, worker processes in background, frontend polls status
  • Hybrid threshold — configurable per team (e.g., docs under 8k tokens → full inject, over → RAG)
  • Embedding model — configurable per team alongside their chat model (e.g., OpenAI text-embedding-3-small)
  • OCR hook — if file is image or scanned PDF, worker calls external OCR microservice before parsing
  • Documents are scoped to team — users in the same team can share docs within conversations
  • Conversation history trimming — oldest messages trimmed first, always keep system prompt + last N turns
